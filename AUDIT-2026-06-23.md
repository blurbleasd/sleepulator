# Sleepulator — code audit (2026-06-23)

A blind, ground-up read of the Swift app: the full audio/state/timer core read line-by-line,
plus a parallel deep read of the views, ambient scenes / Metal shaders, and
parsers / persistence / tests / widget. Findings were verified against the actual source before
being recorded here — no reliance on prior docs or memory.

**Method:** every High-severity claim was confirmed firsthand in the code (the dead proxy, the
Hashable/Equatable contract break, the off-main `updateParams` path).

**Severity legend:** 🔴 ship-blocking / data-or-stability risk · 🟠 correctness · 🟡 quality/perf · ⚪ nit

---

## Top-line assessment

The real-time audio engine is genuinely well-built — a lock-free atomic double-buffer guarded by a
DEBUG `dispatchPrecondition(.onQueue(.main))`, per-layer denormal-flushed DSP with inactive-layer
early-out, the `MTAudioProcessingTap` retain/release dance, the `isLoadingItem` race guard, and
KVO-observer-before-guard ordering. The problems cluster in three places:

1. **Audio-session callbacks ran off the main thread** → could trip the engine's own DEBUG
   precondition and violate the lock-free buffer's single-writer invariant on a headphone unplug.
2. **The "freeze the visuals to save battery" optimization rarely fired** — it was gated behind a
   running sleep timer, so a no-timer sleep session animated at full rate.
3. **A configured network proxy (`feedProxyUrl`) was silently dead** — persisted + shown in
   Settings but never applied to any fetch.

## Findings

### 🔴 / 🟠 Concurrency & correctness

| # | Where | Issue | Status |
|---|-------|-------|--------|
| 1 | `Services/AudioSessionController.swift` | `onInterruption` / `onRouteChange` / `onAppBackground` forwarded on the notification's **posting thread (not main)**; only `onOnlineChanged` hopped to main. Route change → `syncBeatMode` → `setBeatMode` → `updateParams`, which carries `dispatchPrecondition(.onQueue(.main))` + a single-writer invariant on the lock-free param buffer. A headphone unplug delivered off-main would trap in DEBUG and race the buffer in release; `@Published` reads/writes also happened off-main. | **Fixed** — all three forwards now hop to main. |
| 2 | `Models/Models.swift` | `Episode` had a custom `==` on `id` only but kept a **synthesized `hash(into:)` over all fields** → equal values with unequal hashes (broken `Set`/dict contract). `Podcast` hashed its entire `[Episode]` array. | **Fixed** — both are now `id`-based for `==` and `hash`. |
| 3 | `Services/SleepTimerService.swift` | `tick()` (and `PomodoroService.tick()`) read `kind`/`sleepTimerEnd` (`phaseEnd`) on the GCD timer's global queue while they're written on main — a data race. | **Fixed** — state is now read inside the main hop; only `Date()` is captured off-main. |

| # | Where | Issue | Status |
|---|-------|-------|--------|

## Known-good (don't regress)

- Real-time DSP: lock-free atomic double-buffer + DEBUG single-writer precondition, per-sample gain
  smoothing with denormal flush, inactive-layer early-out, decorrelated per-layer RNG seeds,
  `engine.stop()` before freeing render-thread buffers in `deinit`.
- `PodcastPlayer`: tap retains the player and balances it in `finalize`; `isLoadingItem` blocks
  trailing-tick position poisoning; adaptive rewind on resume.
- Sleep timer: in-process fade + RMS/AVPlayer keep-alive **and** an out-of-process notification
  backstop **and** `reconcileIfExpired()` on foreground — three independent nets, once-only `didFire`.
- `AudioDownloader`: App Support (not Documents) + `isExcludedFromBackup`, free-space pre-flight,
  pure/testable LRU eviction.
- Accessibility across the views is unusually thorough.

---

## Deferred (need device verification or larger scope)

- **Shader `float time` precision (H4):** the 32-bit `time` fed into per-frame dither/twinkle hashes
  in `*Shader.metal` quantizes after hours, weakening the anti-OLED-banding dither. Needs a
  shader-side wrapped-phase change + on-device A/B. (The CPU/Canvas scenes use `Double` and are fine.)
- **Idle-based scene freeze without a timer:** the additive `scenePhase`/AOD gate above covers
  background + Always-On; fully decoupling freeze from the sleep timer (e.g. an idle timeout) is a
  bigger UX change worth doing deliberately.
- **Chrome flags** (`ambientScreensaver`/`screenDimmed`/`playbackNote`) still live on the engine.
  They're read only by views that already legitimately observe it (ContentView/HomeView/MiniPlayer/
  NowPlayingSheet), so splitting them off is high-plumbing / low-reward — deferred.
- **`NowPlayingSheet` 250pt hero art** still uses raw `AsyncImage` — its palette-tinted placeholder
  differs from `CachedAsyncImage`'s gray, so it was left as-is to avoid changing that prominent surface.
- **`PersistenceMigrator`** still has no test coverage — its `run()` mutates shared `UserDefaults` +
  the file store, so a clean unit test needs fixture isolation first.
- **Shader `float time` precision** and **idle-based freeze without a timer** remain (see above).

## Tab-root over-observation (done)

`LibraryView` was re-running its episode-filtering body on *every* engine publish (podTitle on each
episode advance, transport, settings). It only reactively needs `isOnline` + `finishedEpisodes` and
only calls `queueManager.playEpisode`, so:

- Added a tiny `Connectivity` child (`@Published isOnline`) on `AudioEngine`, fed by the network
  monitor; `audio.isOnline` is now a non-published passthrough.
- `LibraryView` (and `AddPodcastSheet`) now observe `queueManager` + `connectivity` and hold `audio`
  as an unobserved `let` (only to pass to the pushed detail/add views). The podcast list is now
  immune to unrelated engine churn — it re-renders only on queue / finished / network changes,
  exactly what it displays.
- `PodcastDetailView` takes `connectivity` for its offline banner (it stays a full `audio` observer —
  it's a pushed detail view, not a tab root).

`SettingsView` got the same treatment via a `PlaybackSettings` child:

- The 7 comfort/playback settings it binds (`skipInterval`, `stereoWidth`, `nightLimiter`,
  `limiterByMode`, `sleepEQ`, `sleepEQIntensity`, `beatRouting`) moved off `AudioEngine` into a
  `PlaybackSettings` child. Each setter persists to UserDefaults and applies its side-effect to the
  live audio path via an `on*` callback the engine wires (the logic that used to live in the engine's
  `@Published` didSets).
- `AudioEngine` keeps **computed passthroughs** (`audio.skipInterval`, …) so every other reader
  (NowPlayingSheet, MiniPlayer, lock-screen glyphs, `applyLimiterForMode`, `isochronicActive`,
  restore) is unchanged.
- `SettingsView` now observes `queueManager` + `PlaybackSettings` and holds `audio` as an unobserved
  `let` (used only for `reloadAfterRestore`). Its two queue-backed toggles bind `$queue.*` directly.

`HomeView` keeps observing the engine — it *is* the audio-control surface, not over-observation.
56/56 tests green; build clean. **Caveat: SwiftUI re-render behavior (offline banner updates, tab bar
hides on idle, and — for SettingsView — the limiter/EQ/beat-routing toggles actually applying to live
playback) is not unit-testable. The limiter/EQ path is exactly the engine work the project gate says
needs a real-device run; verify the toggles still take effect on hardware.**
