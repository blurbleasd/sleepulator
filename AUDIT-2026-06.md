# Sleepulator — Blind Code Audit

_Date: 2026-06-21 · Scope: native SwiftUI app under `Sleepulator/` (~5,700 LOC). Static review only — not compiled or run on device. The archived React PWA (`archive_webapp/`) was not audited._

The app is in good shape: the audio architecture is thoughtful, the hard iOS problems (interruptions, route changes, background keep-alive, declick/zipper smoothing, the dual generative-engine + AVPlayer split) are handled with real care, and the comments explain the non-obvious decisions well. The findings below are mostly polish, robustness, and pre-submission items rather than things that are visibly broken.

Severity key: **P0** = correctness / App Store blocker · **P1** = real bug or battery/perf cost users feel · **P2** = robustness · **P3** = hygiene.

---

## P0 — Correctness & submission blockers

### 1. Podcast descriptions are silently dropped for most real feeds (CDATA)
`PodcastParser` collects `description` / `content:encoded` only via `foundCharacters` (PodcastParser.swift:91-111). `XMLParser` delivers CDATA-wrapped text through a separate `parser(_:foundCDATA:)` delegate method, which isn't implemented. The large majority of real podcast feeds CDATA-wrap their show notes (and sometimes titles), so the expandable episode description in `EpisodeRowView` will be empty for them.
**Fix:** implement `parser(_:foundCDATA:)`, decode the bytes as UTF-8, and append to the same `current*` buffer the active element points at. Add a unit test with a CDATA fixture.

### 2. Downloaded episodes live in the Documents directory
`AudioDownloader.getLocalUrl` writes cached audio into `.documentDirectory` (AudioDownloader.swift:9-13). That directory is iCloud/iTunes-backed and (if the app ever sets `UIFileSharingEnabled`) user-visible. Apple's data-storage guideline (2.5.x) says re-downloadable content must not be backed up. There's also **no cache-size cap or eviction** — downloads accumulate until the user manually deletes or enables "delete on completion," so a heavy user fills their disk.
**Fix:** store under `.cachesDirectory` (or App Support) and set `URLResourceValues.isExcludedFromBackup = true` on the file. Add an LRU/size-budget sweep.

### 3. Privacy manifest is probably incomplete for submission
`PrivacyInfo.xcprivacy` declares only the UserDefaults required-reason API (`CA92.1`). The app reads file existence/attributes (`AudioDownloader`, `StorageManager`) and very likely trips **File Timestamp (C617.1)** and possibly **Disk Space (E174.1)** required-reason categories. The **widget target has no privacy manifest at all**.
**Fix:** audit required-reason API usage and add the missing reason codes; add a manifest to the widget target if it touches any.

### 4. Broad ATS exception
`Info.plist` sets `NSAllowsArbitraryLoadsInMedia = true` (cleartext media from any host). Defensible for an arbitrary-RSS app (many enclosures are still `http://`), but App Review may ask for justification.
**Fix:** keep it if needed, but be ready to justify; scope to specific hosts only if feasible.

## P1 — Real bugs & battery/perf

### 5. `rmsPower` re-renders the whole Home screen ~20×/second, all night, for nothing
`AudioEngine.rmsPower` is `@Published` and updated at ~20 fps from the RMS tap (AudioEngine.swift:106, 254), but **no view reads it** (confirmed: only its own declaration + the assignment exist). Every update still fires `objectWillChange`, invalidating `HomeView` and every child holding `@ObservedObject var audio` 20 times a second for the entire sleep session. This is exactly the re-render storm UI-UX-V3 §5 wanted to avoid, and it's pure battery cost on a screen-locked all-night app.
**Fix:** if the orb pulse no longer needs RMS, stop publishing it (make it a plain property or remove it). If a future visual needs it, move it to a tiny dedicated `ObservableObject` so only that leaf view re-renders.

### 6. iTunes search fires on every keystroke and races
`AddPodcastSheet`'s `onChange(of: searchQuery)` launches a new `Task` per character with no debounce and no cancellation (LibraryView.swift:296-312). Typing a query spawns many concurrent requests; whichever finishes last wins, so results can flicker or end up stale (a fast early request overwriting a slower late one).
**Fix:** debounce (~300 ms) and cancel the in-flight task before starting a new one; ignore results whose query no longer matches the field.

### 7. Per-row synchronous disk decode while scrolling episodes
`EpisodeRowView.onAppear` calls `StorageManager.shared.load("positions.json")` — a full JSON decode of the positions map — for **every row that appears** (LibraryView.swift:572). Scrolling a long episode list re-decodes the same file repeatedly on the main thread.
**Fix:** load positions once in the parent (`PodcastDetailView`) and pass the lookup down, or cache it in memory.

### 8. Lock-screen play/pause can desync
Both `playCommand` and `pauseCommand` are wired to `toggle()` (PodcastPlayer.swift:323-330). When the system or CarPlay sends an explicit *play* while already playing (common after route changes), `toggle()` pauses — the opposite of the request.
**Fix:** `playCommand → resume()`, `pauseCommand → pause()`; keep `togglePlayPauseCommand` for the headset button.

### 9. Sleep-timer `tick()` can run concurrently and double-fire the terminal stop
`tick()` is driven by the GCD repeating timer *and* called as `backgroundTick()` from both the RMS callback and the AVPlayer time observer (SleepTimerService.swift:42-61). Several threads can read `sleepTimerEnd` and each dispatch a main-queue closure that, at expiry, calls `stopAllFn()` then `cancelTimer()`. The stop can fire more than once and `timerRemaining` can jitter.
**Fix:** guard the terminal transition (e.g. nil-out `sleepTimerEnd` atomically / a `hasFired` flag) so stop runs exactly once.

## P2 — Robustness

### 10. Silent `try?` on every engine start / session activation
`engine.start()`, `session.setActive(true)`, and friends are all `try?` with no logging, retry, or user signal (GenerativeAudioEngine.swift:333-334, 394-395; AudioEngine.swift:359-360; PodcastPlayer.swift:248). If activation loses a race (e.g. just after an interruption), the all-night bed can silently fail to (re)start with zero feedback. This is the single highest-value place to add resilience, given the product is "play reliably all night."
**Fix:** log failures, retry once after a short delay, and surface a non-destructive `playbackNote` if start still fails.

### 11. Real-time safety relies on undocumented lifetime invariants
The generative engine's double-buffered `AudioRenderParams` uses a plain `Int` read index with no atomics/memory barrier (GenerativeAudioEngine.swift:340-347), and the MTAudioProcessingTap holds the `PodcastPlayer` as `Unmanaged.passUnretained` (PodcastPlayer.swift:411-440). Both are *fine today only because* both objects live for the whole app lifetime. If either ever becomes recreatable, you get torn reads / use-after-free in the finalize callback.
**Fix:** add a comment stating the "lives forever" invariant explicitly, and consider an atomic index (`OSAllocatedUnfairLock`-free swap or `stdatomic`) for the param hand-off.

### 12. Unbounded growth in a couple of stores
`finishedEpisodes` is a `Set<String>` of every episode ever finished, persisted to UserDefaults forever (PodcastQueueManager.swift:27-32). Positions are capped at 100 but finished-IDs aren't. Over months this bloats UserDefaults.
**Fix:** cap or prune (e.g. keep only IDs still present in the library/queue).

### 13. Leaked observers if `AudioEngine` were ever re-created
The block-based `NotificationCenter` observers for `StartSleepulatorMix` / `SetSleepulatorTimer` (AudioEngine.swift:326-342) store no tokens and are never removed, and `NWPathMonitor` is never cancelled. Harmless for a single app-lifetime instance, but a latent leak/duplicate-handler bug if that assumption changes (e.g. previews, tests).
**Fix:** keep the observer tokens and remove them in `deinit`; `monitor.cancel()` in `deinit`.

### 14. Backup/Restore is restart-gated and unvalidated
Import writes raw JSON straight into the backing files/UserDefaults and tells the user to relaunch (SettingsView.swift:229-270). No schema/shape validation — a malformed backup can write undecodable blobs (recovered only because `load` returns nil). It also blind-writes any unknown key into UserDefaults.
**Fix:** validate each section decodes before committing; whitelist keys; reload state in-process instead of requiring a restart.

### 15. Synchronous disk reads during launch
`StorageManager.load` is synchronous and is called on the main thread in the inits of `AudioEngine`, `PodcastPlayer`, and `PodcastQueueManager` (queue.json, positions.json, mixes.json, library.json). Files are small so it's minor, but it's all on the launch path.
**Fix:** acceptable for now; if launch feels slow, defer non-critical loads.

## P3 — Hygiene

- **16. `CLAUDE.md` is stale.** It documents the now-dead React/Vite PWA (AppContext.jsx, MixBus.js, Vitest, the removed "Sleep Safe" Render proxy) and will actively mislead any agent. Rewrite it for the Swift app, or at minimum add a "now native; webapp archived in `archive_webapp/`" banner.
- **17. Spec contradiction.** `UI-UX-V3-SPEC.md` §6 still calls it the "Sleep Safe Audio Limiter"; the code and `AUDIO-LIMITER-SPEC.md` renamed it "Night Limiter." UI-UX-V3 is the stale one.
- **18. Dead code.** `HeaderBar` and `HeroTransport` (HomeView.swift:404-671) are fully defined but never instantiated — the live layout uses `ModeSwitcher` + `OrbButton` + `MixDrawer`. Delete them.
- **19. Committed per-developer files.** `UserInterfaceState.xcuserstate` and `xcschememanagement.plist` are tracked despite being gitignored (they predate the rule). They churn constantly and leak the `melpools` username. `git rm --cached` both.
- **20. Uncommitted WIP in the working tree.** The "Limiter follows mode" feature (`limiterByMode`) spans `AudioEngine.swift` + `SettingsView.swift` and is sitting uncommitted. It looks complete and coherent — commit or stash it so it isn't lost.
- **21. Deployment target.** App + widget are `IPHONEOS_DEPLOYMENT_TARGET = 17.6`, tests `17.0`. 17.6 is a high floor with no obvious API need; consider dropping to 17.0 for wider device support, and align the test target.
- **22. Test gaps that are cheap to close.** The pure, regression-prone bits — `PodcastParser` date/duration/CDATA parsing and `NoiseType.migrate` (the green→brown / white→pink / forest→rain map the docs say "must exist or saved mixes break") — have **no tests** despite being trivially unit-testable. Also `AudioStateTests.testPositionPruneCapsAt100` writes to the real shared `StorageManager.shared` singleton with a 1-second wall-clock wait, making it slow and order-dependent.
- **23. Stray `print()` debugging.** Several `print()` calls remain in the audio/network paths (e.g. PodcastPlayer.swift:314, LibraryView.swift:241, SettingsView.swift:274). Use `os_log` or strip for release.

## Suggested order of attack

1. **Quick wins, high value:** #5 (kill the 20 fps re-render), #6 (debounce search), #8 (lock-screen play/pause), #16/#19/#20 (docs + git hygiene). All small, all felt immediately.
2. **Before next App Store submit:** #1 (CDATA descriptions), #2 (downloads out of Documents + size cap), #3 (privacy manifest), #4 (ATS justification).
3. **Reliability pass:** #9 (timer double-fire), #10 (engine-start resilience), #7 (per-row disk read).
4. **When touching those subsystems:** #11–#15, #21–#23.

> Caveat carried over from the project's own docs: the audio engine's correctness genuinely can't be proven by the unit tests (no real `AudioContext`/RT thread in XCTest). Anything here touching the engine, session, or limiter still needs the on-device, installed-as-PWA… er, installed-app, screen-locked verification described in `TESTING.md`.
